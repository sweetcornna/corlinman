## Synthesized implementation plan

(Synthesis agent hit a transient API error; this section authored from the two
structured component designs below.)

**Guiding stance:** every content-mutating stage is behind a config flag that
defaults **OFF**; stock behavior is unchanged. Two tracks, lowest-risk first.
The persona body is **never auto-rewritten by default** (explicit human approval),
and the built-in `grantley` (`is_builtin=1`) is **blacklisted** from rewrite.

### Track L — complete the (skill/general) loop

| Phase | Component | Risk | Flag (default off) | Live mutation? |
|---|---|---|---|---|
| **L1** | Observer bootstrap (`gateway/evolution/__init__.py` + HookBus hoist in `entrypoint.py`) | med | `[evolution.observer].enabled` | no (writes signal rows) |
| **L2** | Engine `run_once` default scheduler job (mirror `evolution.darwin_curate`) | med | `[evolution.engine].enabled` | no (mints *pending* proposals) |
| **L3** | ShadowTester console script + in-process job | med | `[evolution.shadow].enabled` | no (tempdir sandbox only) |
| **L4** | Real per-kind apply `KindHandler`s on the gateway applier (skills/prompt-template files) | **high** | `[evolution.apply].real_mutations` | **yes** (atomic file write, real inverse_diff) |
| **L5** | In-process `AutoRollbackMonitor` tick + `watched_event_kinds` for non-memory_op | **high** | `[evolution.auto_rollback].enabled` | reverts within grace window |

L1–L3 are safe (no live target is ever written) and can ship first. L4 is the
first real-mutation phase (skills only — persona explicitly excluded); L5 arms
auto-revert. Apply/rollback plumbing is built **persona-agnostic** so the P track
slots in later without re-touching the seam.

### Track P — persona body as an evolution target (highest-gated, all behind `[evolution.persona_review].enabled`)

| Phase | Component | Risk |
|---|---|---|
| **P1** | `persona_quality` scorer + curator (rubric over `system_prompt` + life-state coherence) → `persona.quality.issue` signals | med (read-only) |
| **P2** | `PersonaReviewHandler` propose-side (kind `persona_review`, **excluded from default `enabled_kinds`**) | high |
| **P3** | `PersonaSimulator` + persona eval set (shadow, tempdir-only, before≥after rubric gate) | high |
| **P4** | `PersonaApplyHandler` on the gateway applier — full prior-row snapshot → `PersonaStore.update`, `is_builtin` refused | high |
| **P5** | Apply policy: **explicit approval default**; optional high-confidence auto-apply behind a 2nd flag (`auto_apply`, needs shadow pass_rate=1.0 + rubric gain + no new orphan placeholders + live auto-rollback) | high |

### Recommended sequence & gate
Ship **L1→L2→L3** first (passive, reviewable, no live mutation). Treat **L4, L5,
and all of P** as separate, individually-approved changes — each is irreversible-ish
and high-risk. **Do not** enable any persona-rewrite path on a deployment running
the real `grantley` without the blacklist + explicit-approval gates verified.

### Open questions to resolve before implementing (from the design)
1. HookBus hoist ordering + **who actually emits hook events** — without a live producer the observer records few signals beyond darwin's.
2. In-process scheduler builtins vs the example subprocess crons (avoid double-runs).
3. Which applier becomes canonical — flag-selected vs replacing the store-only one in the admin route.
4. ShadowTester eval-set location + which `KindSimulator`s ship enabled.
5. `watched_event_kinds` target-string normalization (`skills/<name>.md` vs bare name) or the monitor never sees a breach.

## Persona-as-evolution-target design (components)


_Design adds a NEW highest-gated, opt-in, reversible persona-body evolution target that reuses the existing evolution framework end-to-end rather than building a parallel pipeline. The persona body is the personas.sqlite system_prompt (mutated via corlinman_server PersonaStore.update, store.py L309); 'life-state coherence' is checked against the separate corlinman_persona agent_state.sqlite life-state document (PersonaState.state_json['life'], mood, companions). The pipeline mirrors the darwin (SKILL.md) loop one-for-one: a persona.quality.issue signal source (rubric scorer + curator cloned from DarwinScorer/darwin_curator) -> a PersonaReviewHandler that clusters signals and mints high-risk EvolutionProposals (engine-side KindHandler, like DarwinHandler) -> a PersonaSimulator in the shadow sandbox that applies the patch to a tempdir copy and re-scores (KindSimulator, like SkillUpdateSimulator) -> a GATED apply through the gateway's pluggable KindHandler-map EvolutionApplier (gateway/evolution/applier.py, which already has the apply/revert+MutationOutcome/inverse_diff machinery but no handlers) -> rollback by restoring the prior-row snapshot captured in EvolutionHistory.inverse_diff, driven by the AutoRollbackMonitor once watched_event_kinds gains a persona regression contract. Every content-mutating stage is gated behind a single master flag [evolution.persona_review] enabled (default false), the kind is deliberately omitted from the default EngineConfig.enabled_kinds, risk='high' forces shadow before approval, apply defaults to explicit human approval (auto-apply is a second opt-in flag requiring perfect shadow + rubric improvement + a live auto-rollback arm), and builtin personas (grantley) are blacklisted from rewrite. Wiring follows the proven default-off persona.life_advance scheduler-integration template exactly._
