- **R4-F3 (feature — background subagent dispatch):** `subagent.spawn run_in_background:true` returns `run_in_background_not_implemented` (safe, honest). Docs aligned to reality in Round 4 (`docs/multi-agent.md` "Background dispatch"); the safe rejection is kept. A real implementation is a scoped multi-part feature (deliberately NOT shipped as a misleading pure-LLM MVP — analysis in `audit/evidence/round-4/F3-bg-subagent/analysis.md`, contract research in workflow `wf_6f13daa8-3f5`). Required pieces:
  1. **`SubagentRequest` schema** (`system/subagent/store.py:87`): add `tool_schemas`/allowlist snapshot, `model`, `parent_depth`, `max_wall_seconds` — captured at `_dispatch_via_background` time (`tool_wrapper.py:846`). Without the parent tool snapshot a background child can only run pure-LLM (forwarding a `tool_allowlist` with `parent_tools=None` auto-rejects as escalation, `runner.py:167-176`). **Product decision required**: what tool allowlist may an autonomous *background* agent use (esp. `run_shell` / fs-write)?
  2. **Real `run_child_factory`** (`entrypoint.py` ~1671, replace `_unwired_run_child_factory`): a LAZY closure over `state` + `resolved_data_dir` (NOT value-capture — `state.provider_registry` is still None at construction; the providers bootstrap runs later). At invocation: `card = registry.get_or_default(req.subagent_type)` (→ REJECTED if None); `model = card.model or state.config['models']['default']` (never `''` → KeyError); `provider = state.provider_registry.resolve(model, aliases=...)` (NOT the specs-less module `resolve`); reconstruct `ParentContext` from req; `run_child(..., parent_tools=<per policy>)` wrapped in `asyncio.wait_for(ceiling)` + try/except → ERROR `TaskResult`.
  3. **Thread the dispatcher** into `dispatch_subagent_spawn` at `agent_servicer.py:1865` (else background stays at the sentinel even with a real factory).
  4. **Journal notification**: add `start_turn_for_subagent_notification` to the real `AgentJournal` backends (missing everywhere → the dispatcher's terminal "synthetic user msg" is a no-op, so the parent never sees the result). Without this the feature is near-useless.
  5. **Supervisor routing** so all three R3-004 caps (depth / per-parent concurrency / tenant) apply — calling `run_child` directly bypasses two of them.
  6. **Boot reconciliation**: flip pre-existing `running` rows to `stalled` on startup (no sweep exists → orphaned rows after restart permanently consume the tenant cap, `store.py:74`).

There is NO `GET /wstool/runners` (or any runner-registry) HTTP route in the gateway — `grep -rn "wstool/runners" python/` is empty. The page used to poll `fetchRunnersMock()` (`ui/lib/mocks/nodes.ts` → `return []`) every 5s and render the generic "No runners registered" empty state, falsely implying a working-but-idle registry; the detail-drawer "Reconnect" was an inert no-op (`ui/components/nodes/node-detail-drawer.tsx:196`). Honest-aligned in R5: the page now renders an explicit "not yet available (no backend endpoint)" panel and polls nothing (`ui/app/(admin)/nodes/page.tsx` + `NotImplementedBlock` in `ui/components/nodes/offline-block.tsx`). The full topology / side-rail / detail-drawer UI and the `Runner` type/`summariseRunners` mock are retained so they can be re-wired with no UI rework. Required pieces for the real fix:
  1. **Gateway HTTP route** — expose `corlinman-wstool`'s in-memory registry. The data already exists on `ServerState` in `python/packages/corlinman-wstool/src/corlinman_wstool/registry.py`: `runner_count()` (`registry.py:132`) and `self.runners` (`runner_id -> ConnHandle`, each with `.tools`); `advertised_tools()` (`registry.py:128`) gives the tool→runner index. Add a read-only endpoint e.g. `GET /v1/wstool/runners` returning a serialised snapshot (per-runner: id, advertised tool names, connection age, last-ping/health if tracked) plus an aggregate count. Map the existing `ConnHandle` fields onto the UI `Runner` shape (`ui/lib/mocks/nodes.ts`: id/hostname/ring/health/latencyMs/toolCount/connectedForSec/lastPingMs/errorRate/tools) — several UI fields (ring/slot/latency history/errorRate) have no server-side source yet, so either add lightweight server-side bookkeeping or drop those UI affordances.
  2. **Auth + wiring in entrypoint** — register the route behind the same API-key dependency the other gateway routes use, and pass it the live `ServerState`/registry handle wired at WS-tool-server bootstrap (the route must read the SAME registry instance the WS server mutates, not a fresh one). Add to the gateway's route table in `entrypoint.py` next to the other `/v1/...` mounts.
  3. **UI: point the query at the real endpoint** — in `ui/app/(admin)/nodes/page.tsx` restore a `useQuery<Runner[]>` whose `queryFn` calls `apiFetch("/v1/wstool/runners")` (replacing the deleted `fetchRunnersMock` poll), normalising the server payload to `Runner[]`. Re-mount the retained `PageHero`/`StatsRow`/`CapabilityFilter`/`TopologyGraph`/`NodeSideRail`/`NodeDetailDrawer`/`EmptyBlock`/`OfflineBlock` (the loading/empty/error branch logic from the pre-R5 version, preserved in git history) and remove `NotImplementedBlock` from this page. Decide on refresh cadence (poll interval or SSE — `registry.py` would need a change/notify hook for true streaming).
  4. **Reconnect action** — implement `onReconnect` (currently `undefined`, so the button is disabled via `disabled={!onReconnect}` at `node-detail-drawer.tsx:197`): a gateway POST that signals the WS-tool server to drop+await re-handshake for a given `runner_id`, or remove the affordance if out of scope.   5. **i18n strings** — the R5 honest panel copy is intentionally inlined in `NotImplementedBlock` (no existing translation string truthfully says "no backend endpoint"); when the real feature lands, fold any retained copy back into `ui/lib/locales/en.ts` + `zh-CN.ts` under the existing `nodes.tp.*` namespace.

- **C4-EMB (feature — embeddings / dense retrieval is DEAD):** the `[embedding]` config block is parsed, seeded by onboarding, and surfaced via `/admin/embedding`, but nothing constructs an embedder at runtime — so semantic/dense retrieval does **not** work. Docs aligned to reality in Round C4 (`docs/providers.md` §1 "Not wired yet" note + the three recipe captions). Evidence: every provider `.embed()` raises `NotImplementedError` (`openai_provider.py:281`, `anthropic_provider.py:405`, `google_provider.py:178`); episodes retrieval is recency + BM25 only and the stored embedding BLOB is never read for a cosine path; the `corlinman_embedding.EmbeddingRouter` the bindings nominally target does not exist (no `python/packages/corlinman-embedding/` package). Required pieces:
  1. **A real `embed()` impl** for at least one provider kind (OpenAI `/v1/embeddings` is the obvious first — replace the `NotImplementedError` at `openai_provider.py:281`; batch + dimension from `[embedding].model`/`dimension`).
  2. **Wire `register_embedding_provider_factory`** (already exported by `corlinman_episodes.cli`, `cli.py:127`) in gateway boot (`lifecycle/entrypoint.py`): construct an `EmbeddingRouter`-shaped callable from the resolved `[embedding]` binding (`provider` → `ProviderRegistry` entry, must be embedding-capable) and hand its bound `embed` to the episodes CLI/resolver. Today the factory is never registered, so the embed seam stays `None`.
  3. **Episodes resolver cosine path**: read the stored embedding BLOB and add a cosine (or RRF-with-dense) branch in the episodes retrieval ranker, gated on the registered embedder being present; fall back to BM25 + recency when absent. Without this the embedding column written at ingest is dead weight.
  4. **Provider capability gating**: keep the `kind`-level embedding capability flags honest so `/admin/embedding` + `<DynamicParamsForm>` only offer embedding-capable providers once (2) lands.

- **C4-PLUGIN-ASYNC (feature — plugin async/deferred callback is a permanent no-op):** `plugin_type = "async"` accepts a `task_id` but the deferred-result path is not connected. Docs aligned to reality in Round C4 (`docs/plugin-authoring.md` §5 banner + type table + timeout FAQ). Evidence: `POST /v1/plugins/callback/{task_id}` returns 501 unconditionally because `GatewayState.plugin_async_tasks` is never assigned anywhere (no `plugin_async_tasks =` in src or tests; the route's `None` guard at `routes/plugin_callback.py:45` always fires); `plugin_invoker` never parks `accepted_for_later` on a registry — it just surfaces the `task_id` verbatim to the model (`grpc/plugin_invoker.py:198`). Required wiring:
  1. **Construct + assign the registry**: build an `AsyncTaskRegistry` (`corlinman_providers.plugins.async_task:42`) in `lifecycle/entrypoint.py` and assign it to `GatewayState.plugin_async_tasks` (the slot at `routes/register.py:78`); `register.py:123-132` already mounts the callback router only when that slot is a real `AsyncTaskRegistry`.
  2. **Park on accept**: in `plugin_invoker` `accepted_for_later` branch (`plugin_invoker.py:198`), call `registry.register(task_id)` to create the awaitable future *before* returning to the loop, and have the reasoning loop await it (with the manifest `timeout_ms` ceiling + cancellation) instead of treating `accepted` as a terminal string.
  3. **Resolve on callback**: the `/v1/plugins/callback/{task_id}` route resolves the parked future from the registry with the posted `{status,result}` payload (one-shot, then evict); unknown/expired `task_id` → 404/410, not 501. Keep `task_id` as the one-time bearer-less credential the docs describe.
